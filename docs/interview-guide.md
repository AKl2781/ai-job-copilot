# 面试问答

## 30 秒项目介绍

AI Job Copilot 是我完成的一个本地 AI 求职助手 MVP，采用 Manifest V3 浏览器扩展加 FastAPI 后端。扩展读取 JD 并保存候选人资料；Windows + Edge 下通过白名单 Native Messaging 按需启动本地后端，点击一次分析即可等待健康检查并继续请求。后端隔离 DeepSeek API Key，让模型提取证据，再按固定权重计算可解释分数。项目有 pytest 与 Node 测试，并完成过真实本地启停验收。

## 2 分钟项目介绍

项目解决的是求职时反复复制 JD、对照个人技能和整理沟通文案的问题。我把它拆成扩展端和后端两层：扩展负责获取当前页面、保存候选人资料和展示结果，后端负责模型调用、配置隔离、数据校验与异常处理。

页面提取不是简单读取整个 `body`。它先看用户有没有选中文字；没有的话，会从岗位语义选择器和 `div/section/article/main` 中收集候选节点，根据岗位关键词、文本长度、交互元素占比、节点占页面比例和父子关系打分，再清理部分按钮与尾部推荐职位。没有达到阈值时依次回退到 `main`、`article`、`role="main"` 和 `body`。这样兼顾了自动化和可恢复性。

后端接收岗位标题、JD 和 `candidate_profile`，API Key 只留在后端环境。DeepSeek 被要求返回岗位要求、候选人证据和离散状态，后端忽略模型自由给出的旧 `score`，按核心技能、加分技能、项目、教育和工作经验的固定权重计算总分与 `score_breakdown`。缺少配置、超时、限流、认证、连接和格式错误都有明确处理。当前定位仍是本地 MVP，没有数据库或云端部署，未进行大规模用户验证；评分仅供参考，不承诺全站点适配，也不做自动投递或自动聊天。

## 高频问题

### 1. 为什么做浏览器扩展，而不是普通网页？

核心输入就在招聘页面里。扩展可以在用户主动打开时读取当前 tab，减少复制粘贴，同时保留原页面作为核验依据。当前用 `activeTab`，只在用户触发后获得当前页面权限，比申请所有网站的长期访问权限更克制。

### 2. 为什么采用 Manifest V3？

这是当前 Chromium 扩展的主流规范。页面读取仍基于 popup、`activeTab` 和脚本注入；Windows + Edge 版本增加 service worker，将 `status/start/stop` 白名单消息转发给短生命周期 Native Host，不需要后端开机常驻。

### 3. 岗位详情如何识别？

优先级是：长度足够的选中文字 → 智能候选区域 → `main` → `article` → `role="main"` → `body`。智能候选区域根据正负关键词、文本结构、交互占比、页面占比、语义类名以及父子节点相似度打分，达到阈值后采用得分最高的区域。

### 4. 为什么不为每个招聘网站写选择器？

通用启发式能先覆盖更多未知页面，维护成本较低；选中文字和手动编辑又提供了确定性兜底。代价是准确率不可能在所有网站一致。后续如果基于真实失败样本发现稳定模式，再增加站点适配会更合理。

### 5. 如何减少推荐职位和按钮噪声？

提取时会排除导航、页头页尾、按钮和输入区等祖先节点，计算链接/按钮文本占比，并删除完全匹配的常见按钮行。正文足够长时，遇到“推荐职位”“猜你喜欢”“相似职位”等标题会截断后续内容。

### 6. API Key 如何保护？

Key 只由 FastAPI 后端从系统环境变量或根目录 `.env` 读取，扩展请求里没有 Key。仓库提供的是空值 `.env.example`。这解决了前端直接暴露 Key 的问题，但当前仍是本地开发形态；若上线，还需要密钥管理、鉴权、限流和审计。

### 7. 为什么需要 Pydantic 校验模型输出？

“让模型返回 JSON”不等于返回一定可靠。后端会解析 JSON，再严格校验字段、类型、分数范围，并拒绝额外字段。这样前端可以依赖稳定契约；格式错误会转成 502，而不是把不可控内容继续传递。

### 8. 匹配分数是谁算的？

DeepSeek 不决定最终分数，只提取岗位要求、候选人证据和状态。后端将 matched / partial / unverified / missing 固定映射为 100 / 60 / 25 / 0，并按核心技能 45%、加分技能 15%、项目 20%、教育 10%、工作经验 10% 汇总；不适用维度会重新归一化。`score_breakdown` 给出各维度分数、有效权重、适用性和原因。该分数可复现、可解释，但仍不是客观招聘标准或录用概率。

### 9. 如何避免模型把 JD 要求当成候选人技能？

请求把 JD 与候选人资料明确分区；模型约束要求候选人能力只能来自 `candidate_profile`，未提到的要求应进入待补技能或标记待确认，并禁止把“了解”“学习中”等表述升级为更强熟练度。相关关键约束有测试覆盖，但模型输出仍应由用户核验。

### 10. 异常处理覆盖了什么？

输入空白会被请求模型拒绝；缺少 Key 或配置错误、认证失败、限流、连接失败、超时、上游状态异常和返回 JSON 不合规，都有对应的安全错误。扩展端则显示统一失败提示，并确保按钮恢复可用。

### 11. 测试覆盖了什么？

pytest 覆盖健康接口、三字段请求、确定性评分、Native Host 协议与安全边界、模型 JSON 解析和安全错误。Node 测试覆盖智能岗位提取，以及 popup 的按需启动、防重复请求、新旧响应兼容、自动滚动和复制反馈。发布回归为 44 项 pytest 通过，两个 Node 测试通过；真实 DeepSeek 在线调用未在发布流程执行。

### 12. 当前最大局限是什么？

页面结构差异和上游证据提取的不确定性。通用启发式无法保证覆盖所有站点，固定规则的输出质量仍依赖提取出的要求和证据。另外项目没有账号、服务端存储、生产鉴权和部署，因此只能称为本地 MVP。

### 13. 如果继续迭代，会先做什么？

先收集匿名化的提取失败样本，建立可复现的页面 fixture 与评估集，再决定增强通用规则还是增加站点适配。产品层面会优先补充结果可追溯性和用户确认流程；如果走向线上，再补鉴权、限流、密钥管理、隐私策略和可观测性。

### 14. 项目中最能体现工程能力的部分是什么？

不是单次模型调用，而是把浏览器权限、页面提取兜底、前后端数据契约、Key 隔离、模型输出校验、异常映射和测试串成了可运行闭环，并对当前不能保证的能力做了明确边界说明。

## 表述红线

- 不说“支持所有招聘网站”，应说“通用启发式识别，提供多级回退和手动兜底”。
- 不说“AI 能预测录用概率”，应说“DeepSeek 提取要求与证据，后端按公开固定规则计算辅助分数”。
- 不说“模型结果准确”，应说“结构化辅助判断，需要用户核验”。
- 不说“数据已安全合规”，应说“Key 已后端隔离；生产级隐私、鉴权与审计尚未实现”。
- 不说“已上线”或“生产可用”，应说“本地 MVP；曾完成合并前真实 DeepSeek 联调，合并后的在线联调待手动验收”。
- 不说“自动投递/自动聊天”，项目没有实现这些功能。
